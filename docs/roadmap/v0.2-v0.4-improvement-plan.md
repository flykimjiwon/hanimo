## Part 1: list_files 스마트 처리 플랜

### 현재 문제
- 재귀 모드 500개 하드 리밋
- `_legacy_ts/node_modules/**`만 10,000+ 파일
- LLM이 전체를 보고 싶어해서 같은 호출 반복
- **해결됨(v0.2.1)**: default skip 리스트 확장 + hint 메시지

### 추가 개선안

#### 안 1: Paginated 재귀 탐색 (추천)
```go
func ListFiles(dir string, recursive bool, page int, pageSize int) {
    // page=1 → 1-500
    // page=2 → 501-1000
    // page=3 → 1001-1500
    // 각 응답에 "next_page: 2" 힌트
}
```

LLM 프롬프트:
```
If you see "next_page" in result, you can request more by calling
list_files again with page=2. Only do this if truly needed.
```

#### 안 2: Tree View (폴더만) — **가장 추천**
새 도구 추가:
```go
// list_tree(path, maxDepth int) — returns directory tree structure only
// 파일 목록 X, 디렉터리 구조 O
```

출력 예시:
```
hanimo/
├── cmd/
│   └── hanimo/
├── internal/
│   ├── agents/
│   ├── app/
│   ├── config/
│   ├── llm/
│   │   └── providers/
│   ├── mcp/
│   ├── session/
│   ├── tools/
│   └── ui/
├── knowledge/
│   └── docs/
│       ├── css/
│       ├── go/
│       └── ...
└── docs/
```

장점:
- LLM이 프로젝트 구조를 **한 번에** 파악
- 파일 개수와 무관하게 빠름
- 이후 필요한 디렉터리만 `list_files`로 상세 조회

#### 안 3: Glob-based Smart Listing
```go
// list_files(path, pattern="*.go") — glob 패턴으로 필터링
// list_files("/project", "**/*.ts") → TypeScript 파일만
```

#### 안 4: Size-aware Listing
디렉터리별 파일 개수 먼저 보고 임계값 넘으면 건너뛰기:
```
frontend/      (15,234 files — skipped, use subpath)
internal/      (47 files)
├── agents/    (4 files)
├── app/       (1 files)
└── ...
```

### 채택안: **안 2 (list_tree) + 안 3 (glob filter) 조합**

1. 먼저 `list_tree`로 구조 파악 (빠름, LLM 친화적)
2. 필요한 디렉터리 → `list_files`
3. 파일 패턴 필요 → `glob_search` (이미 있음)
4. 재귀 전체 탐색 → **제거** (사용 금지)

### 구현 우선순위
- 🔥 **v0.2.1**: default skip 리스트 확장 (✅ 완료)
- 🔥 **v0.2.2**: `list_tree` 도구 추가
- ⚡ **v0.2.3**: 재귀 모드를 기본 off로 변경, LLM이 명시적으로 요청해야 사용

#### 2.3 Parallel Execution
```go
// internal/agents/parallel.go

func RunParallel(ctx context.Context, subAgents []*SubAgent) []SubResult {
    results := make([]SubResult, len(subAgents))
    var wg sync.WaitGroup

    // 동시 실행 제한 (예: 3개까지만)
    semaphore := make(chan struct{}, 3)

    for i, agent := range subAgents {
        wg.Add(1)
        go func(idx int, sa *SubAgent) {
            defer wg.Done()
            semaphore <- struct{}{}
            defer func() { <-semaphore }()

            results[idx] = sa.Run(ctx)
        }(i, agent)
    }

    wg.Wait()
    return results
}
```

```
> 이 프로젝트 전체 TypeScript 에러 고치고 테스트까지 통과시켜줘

[Orchestrator] 태스크 분해 중...

📋 4개 서브태스크로 분해:

  └─ SubAgent 4: 최종 빌드 검증

⏳ 실행 중... (3/4 parallel)
  ✅ SubAgent 1: 12 errors found (2.3s)
  ⏳ SubAgent 2: fixing file 5/12 (14.7s)
  ⏳ SubAgent 3: running vitest (8.1s)
  ⬜ SubAgent 4: waiting

[완료 후 Aggregator가 결과 합성]
```

### 사용 시나리오
- **리팩터링**: 파일별 병렬 수정
- **테스트 생성**: 함수별 병렬 작성
- **문서화**: 모듈별 병렬 docstring 생성
- **마이그레이션**: 파일별 병렬 변환

### 구현 우선순위
- 🔥 **v0.3.0**: Orchestrator + SubAgent 기본 구조
- ⚡ **v0.3.1**: Parallel 실행 + semaphore
- ⚡ **v0.3.2**: UX 개선 (progress bar, interrupt)
- 🎯 **v0.3.3**: Depth 제한 + 무한 재귀 방지
- 🎯 **v0.3.4**: SubAgent 간 메시지 전달 (필요 시)

### 위험 요소
- **토큰 폭발**: 4개 SubAgent × 10 iter × 8K context = 320K tokens / 1 요청
- **실패 전파**: 한 SubAgent 실패가 전체 실패로 이어지면 안 됨
- **Race condition**: 같은 파일 동시 수정 방지 (파일 락)
- **디버깅 어려움**: 여러 conversation 동시 추적

### 대책
- 토큰 예산: 전체 SubAgent 합쳐서 max 100K
- 실패 isolation: partial success 지원
- 파일 락: `session.LockFile(path)` API
- 디버그 로그: 각 SubAgent별 로그 파일
